# 扫描范围：全项目、包含路径、排除路径

目标：得到一份**可信的**变更路径集合，作为拆分提交与撰写 message 的唯一事实来源。

## 约定

- 默认使用 `git status --porcelain=v1`（或等价）观察工作区与暂存区。
- **路径均相对于仓库根**。用户可使用绝对路径发言，执行时转换为相对仓库根路径以便命令一致。
- **排除**不是 `git status` 的内置开关；采用「先收集再过滤」或「缩小再收集」组合，见下文。

---

## 模式 A：全项目扫描

```bash
git status --porcelain=v1 -u
```

`-u` 确保未跟踪文件按统一格式列出（与团队习惯一致即可）。

---

## 模式 B：仅扫描指定文件与文件夹（包含列表）

用户给出若干文件或文件夹 `P1..Pn`：

```bash
git status --porcelain=v1 -u -- P1 P2 P3
```

说明：

- Git 会将列出的文件夹展开为其中变更条目。
- 若用户同时给出「包含」与「排除」，建议**先**用 `--` 收窄到包含集合，再应用排除过滤（见模式 C），避免歧义。

---

## 模式 C：在扫描结果上排除路径

典型需求：扫描全项目或某个文件夹，但忽略 `dist/`、某生成物文件夹、或用户点名的子树。

### C1：前缀 / 子路径排除

用户给出排除前缀列表 `E1..Ek`（文件夹建议以 `/` 结尾语义更清晰，实现时规范化）。

对模式 A 或 B 的每一行输出，解析出路径字段后丢弃满足以下任一条件的条目：

- 路径等于某个 `Ej` 或以 `Ej` 为前缀（文件夹排除）。

### C2：Glob 风格排除（可选）

若用户给出 glob（如 `**/*.lock`），在**已解析出的路径字符串**上做匹配过滤。注意：

- glob 语义依赖实现环境；优先让用户用**明确前缀**排除关键文件夹，减少误判。

### C3：排除与暂存区

若某路径已被暂存但落入排除集合：

- 必须在方案中**显式说明**（「该路径当前在索引中」），由用户决定是纳入某次提交、还是先 `git restore --staged` 再按方案重来。**禁止静默改写索引**而不告知用户。

---

## 输出物（交给下一步）

从扫描阶段输出：

1. **最终路径列表**（去重、排序可选）与 porcelain 状态首列含义说明。  
2. **扫描配置摘要**一句话：例如「全仓库，排除 `node_modules/`、`tmp/`」。  
3. 若存在子模块或特殊路径，注明是否纳入本次范围。
